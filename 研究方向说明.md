# ChatREQ研究方向说明

ChatREQ 的研究方向是：面向智能需求工程的知识增强与多智能体协作研究。本方向关注如何从利益相关者的原始表达出发，通过需求工程知识、领域知识、项目上下文和多智能体协作，生成更加完整、明确、可验证、可追踪的软件需求。

## 1. 什么是需求工程

需求工程是软件工程中的基础环节，主要研究如何获取、分析、规约、验证和管理软件系统需求。它关注的不只是“用户想要什么功能”，还包括用户为什么需要该功能、谁会使用该功能、在什么业务场景下使用、受到哪些约束、如何判断需求是否被满足，以及需求变更后如何追踪影响。

一个较完整的需求工程过程通常包括：

- 需求获取：从用户、业务人员、文档、系统环境和实际流程中发现需求。
- 需求分析：识别目标、角色、对象、场景、约束、冲突和优先级。
- 需求规约：将需求整理为结构化、清晰、可检查的文本或模型。
- 需求验证：检查需求是否完整、一致、明确、可验证、可追踪。
- 需求管理：记录需求版本、变更原因、影响范围和追踪关系。

需求工程的难点在于，用户最初给出的表达往往并不完整。用户可能省略业务规则，混合多个目标，使用含糊词语，默认某些领域知识已经被理解，或者在不同场景下表达出相互冲突的期望。

## 2. 什么是智能需求工程

智能需求工程是将自然语言处理、知识表示、机器学习、大语言模型、智能体协作等技术用于需求工程任务的研究方向。它希望提升需求获取、分析、生成、检查和管理的效率与质量。

在智能需求工程中，模型和工具可以辅助完成以下工作：

- 从自然语言中识别需求要素；
- 将口语化表达转化为结构化需求；
- 发现模糊、冲突、遗漏和不可验证表述；
- 根据领域知识补充候选约束；
- 生成用户故事、用例、功能需求和非功能需求；
- 建立需求、知识、场景和设计之间的追踪关系。

但智能化并不意味着可以跳过需求工程本身。相反，越是使用智能工具，越需要明确需求工程规范、知识边界和质量检查机制。

## 3. 为什么大语言模型可以用于需求工程

大语言模型具有较强的自然语言理解和生成能力，能够处理需求工程中大量以文本形式存在的材料。例如，用户访谈记录、业务规则、需求说明、用例描述、系统说明和测试场景都可以作为模型处理对象。

在需求工程中，大语言模型的潜在价值包括：

- 帮助初步整理用户原始表达；
- 从文本中抽取角色、目标、功能动作、对象、约束和场景；
- 将零散表达改写为更规范的需求条目；
- 提出澄清问题，辅助需求访谈；
- 检查需求中的模糊性、遗漏和不一致；
- 生成需求分析报告、用户故事和需求规约初稿。

这些能力适合用于科研训练，但必须放在严格的问题定义和实验验证框架中使用，不能把流畅文本等同于高质量需求。

## 4. 大语言模型在需求工程中可能存在的问题

通用大语言模型在需求工程场景中容易出现以下问题：

- 用户意图理解不足：模型可能只处理表层文字，没有识别用户真正关心的业务目标。
- 需求表达补全过度：模型可能加入用户并未确认的功能、角色或流程。
- 约束遗漏：模型可能忽略时间、权限、数据范围、合规性、性能和异常处理等约束。
- 角色混淆：模型可能混淆管理员、普通用户、审核人员、系统服务等不同角色。
- 场景缺失：模型可能没有说明需求适用的业务场景和触发条件。
- 不可验证：模型可能生成“友好”“高效”“智能”等难以验证的描述。
- 不可追踪：模型可能无法说明生成需求来自哪条原始表达、哪条知识或哪次澄清。
- 意图偏移：模型在扩写过程中可能偏离利益相关者的原始目标。

因此，需求工程中的模型输出不能只看语言是否通顺，还要检查是否保留原始意图、是否符合知识约束、是否能够被验证和追踪。

## 5. 为什么需要引入知识和上下文

需求生成和需求检查离不开知识。缺少知识时，模型只能依靠通用语言模式，容易生成看似合理但缺乏依据的内容。

ChatREQ 关注三类知识：

- 需求工程知识：例如需求完整性、一致性、明确性、可验证性、可追踪性、用户故事格式、用例要素、非功能需求分类等。
- 领域知识：例如医疗、教育、金融、电商、政务等领域中的业务规则、角色关系、数据约束和合规要求。
- 项目上下文知识：例如目标系统范围、已有功能、用户群体、技术限制、历史需求、术语定义和当前版本约束。

引入这些知识的目标不是机械堆叠背景材料，而是在关键任务中发挥作用：帮助模型提出更合适的澄清问题，减少遗漏，识别冲突，约束生成边界，提高需求质量。

## 6. 为什么需要多智能体协作

需求工程本身包含多个子任务。一个单一模型角色很难同时承担访谈、知识检索、需求分析、规约生成、质量检查和追踪归档等任务。多智能体协作的意义在于将复杂过程拆分为职责明确的子任务，让不同智能体承担不同需求工程角色。

在 ChatREQ 中，多智能体不是简单换名字，而是对应具体需求工程任务：

- Interviewer Agent 负责追问与意图澄清。
- Knowledge Agent 负责检索和注入相关知识。
- Analyst Agent 负责识别目标、角色、对象、约束和场景。
- Specification Agent 负责生成结构化需求规约。
- Reviewer Agent 负责需求质量检查。
- Archivist Agent 负责记录原始意图、知识条目和生成需求之间的追踪关系。

多智能体协作可以让需求生成过程更透明，也更容易定位失败原因。例如，如果生成需求缺少约束，可以检查 Knowledge Agent 是否提供了相关知识；如果需求不可验证，可以检查 Reviewer Agent 是否发现了问题。

## 7. ChatREQ试图解决什么问题

ChatREQ 试图解决的核心问题是：如何在大语言模型辅助需求工程时，更好地保持利益相关者原始意图，并通过知识增强和多智能体协作生成更高质量的软件需求。

具体来说，本方向关注：

- 如何识别用户原始表达中的不完整、模糊、隐含和冲突信息；
- 如何分析模型在需求生成中的意图偏移和质量问题；
- 如何组织需求工程知识、领域知识和项目上下文；
- 如何将知识有效注入需求生成过程；
- 如何设计多智能体协作流程；
- 如何检查需求的完整性、一致性、明确性、可验证性和可追踪性；
- 如何将问题、方法、实验和原型转化为论文预研材料与专利交底材料。

## 8. 本科科研训练应重点训练的能力

本科科研训练阶段允许从基础开始，但不能长期停留在“学习中”“了解中”的状态。学生应逐步训练以下能力：

- 概念理解能力：能准确解释需求工程、智能需求工程、知识增强和多智能体协作。
- 问题分析能力：能从案例中识别模糊、遗漏、冲突和意图偏移。
- 文献阅读能力：能围绕研究问题阅读论文，而不是只做摘要。
- 数据整理能力：能构建需求案例、失败案例、知识条目和实验记录。
- 实验设计能力：能设置对比条件、评价指标和人工分析标准。
- 原型实现能力：能完成一个最小可运行模块，用于验证研究机制。
- 科研表达能力：能把问题、方法、实验和结论整理为报告、论文预研材料和专利交底草稿。
