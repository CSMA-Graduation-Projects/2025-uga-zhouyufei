# M01 — ChatREQ 方向理解报告

## 1. 问题背景

软件开发中，需求工程是最容易出问题的环节。用户说出来的需求往往是模糊的、不完整的、甚至自相矛盾的。传统做法是靠需求工程师手工整理，反复跟用户确认，但这个过程非常耗人、耗时，而且严重依赖工程师的个人经验。

随着大语言模型（LLM）的出现，可以用 AI 来辅助需求工程。ChatGPT、Claude 这些通用模型确实能写出看起来很流畅的需求文档。但问题在于，这些输出往往表面流畅、实际有坑：可能漏掉了关键约束、混淆了不同角色的需求、甚至自己编造了用户没提过的功能。这就是"意图偏移"。

ChatREQ 的研究方向就是解决这个问题：如何用大语言模型 + 需求工程知识 + 多智能体协作，把用户零散、模糊的自然语言表达，变成高质量、可验证、可追踪的软件需求？

## 2. 本阶段目标

## 6. 分析

### 6.1 ChatREQ 不是做平台，是做研究
ChatREQ 的核心不是搭一个能用的系统，而是验证"知识增强 + 多智能体协作能不能比裸模型生成出更好的需求"。


### 6.2 多智能体协作为什么比单模型好

单模型一步到位把所有 RE 子任务塞进一次对话，容易注意力分散、信息丢失、无法独立纠错。ChatREQ 拆成 6 个角色各管一块：Interviewer 追问澄清 → Knowledge 知识注入 → Analyst 拆解 8 要素 → Specification 生成规约 → Reviewer 5 维度检查 → Archivist 追踪归档。哪个环节出问题可以独立定位和改进。

与 MA-ARP 论文对比后发现，Reviewer Agent 是 ChatREQ 的差异化优势——论文的方法从 PL 到 VGL 一条线走到底，中间没有独立的质量检查环节。

### 6.3 从 MA-ARP 发现的改进方向

- Analyst 可以拆成并行子智能体：针对不同的建模，有不同的智能体来工作，MA-ARP 证明拆开比一个 Agent 全包效果好
- Archivist 可以从 Agent 角色升级为自动基础设施：MA-ARP 的追溯矩阵比手动记录更可靠，每次的更新，都是追溯矩阵版本的升级
- Reviewer 检查结束后就结束通过了，但是并不能跟用户真满意划等号，可以在reviewer之后加一个轻量验收原型，让用户确认，有反馈我们就回到interviewer
