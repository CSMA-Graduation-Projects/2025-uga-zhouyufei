# M02 — 需求意图案例分析报告

## 1. 任务概述

本报告回答 **RQ1：用户原始需求表达中通常存在哪些不完整、模糊、隐含或冲突的信息？**

本阶段从公开来源（GitHub Issues、知乎、豆瓣、高校留言板、黑猫投诉平台、新闻报道）收集了多条真实或基于真实事件重构的用户原始需求表达，覆盖选课系统，预约系统和电商订单系统三个领域。

## 2. 本阶段目标
对于自然语言的拆解，掌握常见的错误

## 3.使用材料
1. github上的真实案例
2. 知乎上面的回答

## 4.方法
1. 对于8要素粗浅的理解大意，然后去寻找真实案例
2. 用ai生成示例案例学习
3. 根据真实案例去分析这8各要素
4. 最后跟ai一起总结出常见的错误

## 5.结果
1. 回答req1
2. 深入理解8个要素，再需求理解这个大的方面有了更多深入

### 6分析
| **约束遗漏** | 用户未说明操作的条件、限制和规则  |
| **规则不透明** | 系统有隐式规则但用户不知情，直到被惩罚才发现  |
| **系统替用户做决定** | 系统在未通知用户的情况下自动执行不可逆操作  |
| **多目标混淆** | 用户需求中包含多个互相冲突或优先级不清的目标  |
| **边界缺失** | 异常情况和正常情况的边界未定义  |
| **交互反馈不足** | 系统缺少在关键节点的提示和引导 |
| **执行机制缺失** | 规则有但无法落地执行  |


## 7. 分析启示

### 7.1 对 Interviewer Agent 的启示

真实案例的澄清问题不能只问"还有什么没说"。许多问题来自：
- **规则发现**：用户不知道自己不知道什么规则 → Interviewer 需要主动告知相关规则并确认用户是否接受
- **权限边界**：用户假设系统能做某事，但可能涉及公平性争议 → Interviewer 需要把隐含的公平性权衡显式化在重新再交互中告知用户
- **发现**：领域知识（业务规则）在真实案例中覆盖面最广，说明真实用户最常遇到的不是"模型输出写不好"，而是**根本不知道业务规则是什么**，在交互中也可以去告知用户

### 7.2 对 M03（大模型失败案例）的启示

真实案例中「规则不透明」「交互反馈不足」两类问题，直接测试裸 LLM 时可能被模型忽略——模型倾向于假设系统设计是合理的，不会主动质疑规则的不透明性。这恰好可以成为 M03 中验证"意图偏移"的实验素材。
