# MA-ARP 阅读笔记

## 1. 题目

基于多智能体的敏捷需求工程过程模型
(MA-ARP: Multi-Agent Agile Requirements Engineering Process Model)

## 2. 作者与年份

刘芳、涂键、曹舒，东北石油大学计算机与信息技术学院

发表于《计算机系统应用》，2026 年，第 35 卷第 2 期，226–236 页

## 3. 研究问题

敏捷开发中引入需求模型后，需求频繁变更导致模型手动维护成本极高——不及时更新就过时，过时就导致需求工程失败。核心问题：**能不能用多智能体自动维护需求模型？**

论文分解为三个子问题：
- RQ1：多智能体方法能不能完成需求工程任务？
- RQ2：多智能体方法能不能做好需求模型在迭代中的维护更新？
- RQ3：多智能体方法能不能降低需求工程的开销？

## 4. 方法：MA-ARP 模型

整体结构分两块：**人工过程**（头尾，人做决策）+ **多智能体过程**（中间全自动，AI 做执行）。中间靠追溯矩阵贯通。

### 4.1 人工过程（头尾两端）

- **开头**：需求工程师跟用户和利益相关者沟通，收集原始需求表达，提交给多智能体过程
- **结尾**：多智能体过程产出的文档和原型返回给人，拿去跟用户确认
- **中间不做任何人工干预**，全部交给智能体自动处理

核心原则：人做决策（理解用户、确认需求），AI 做执行（分析、建模、生成）。

---

### 4.2 多智能体过程：四层流水线

#### 第 1 层：预处理层 (PL) — 零散需求 → 结构化条目

| 智能体 | 干什么 |
|--------|--------|
| 分类智能体 | 语义理解需求内容，归类为功能需求/非功能需求/约束/业务规则等 |
| 排序智能体 | 给需求排优先级（可集成 MoSCoW、Kano 等多种方法） |

**协作方式**：两个智能体共享一个消息池，分类结果扔进去，排序的来取。不是严格串行，信息流有交叉。

**输出**：结构化需求条目，每一条必须包含——原始需求、需求类型、关联需求、子需求。用「版本号 + 优先级 ID」作为索引键。

---

#### 第 2 层：需求分析层 (RAL) — 核心建模层，最复杂

RAL 包含三个**并行的**建模工作流，每个工作流内部有多个智能体同时干活：

**工作流 1 — 用例图建模（3 个智能体并行）**

| 智能体 | 干什么 |
|--------|--------|
| Actor 识别智能体 | 从需求文本找出「谁在用这个系统」——参与者/角色 |
| UseCase 识别智能体 | 找出「系统要做什么事」——用例 |
| Relationship 识别智能体 | 找出「谁跟谁有关系」——Actor 与 UseCase 间的关联/继承/包含 |

**工作流 2 — 类图建模（4 个智能体并行）**

| 智能体 | 干什么 |
|--------|--------|
| 类识别者 | 找系统里有哪些类/实体 |
| 属性识别者 | 找每个类有什么属性/字段 |
| 函数识别者 | 找每个类有什么方法/操作 |
| 类关系识别者 | 找类之间的继承、关联、聚合、依赖 |

**工作流 3 — 顺序图建模（3 个智能体并行）**

| 智能体 | 干什么 |
|--------|--------|
| 对象识别者 | 找交互中涉及的对象实例 |
| 消息识别者 | 找对象之间发送的消息/方法调用 |
| 消息顺序识别者 | 定消息的先后执行顺序 |

**每个智能体背后都有一套**：专属 Prompt 模板 + 知识库中的模型元素识别规则 + 迭代反馈修正机制。Prompt 示例（Actor 识别）：

> "You are a professional requirement analyst. Please decompose user input and knowledge base rules as follows: 1. Decompose requirements into functional and non-functional requirements... 2. Each requirement must include the following fields... 3. Determine the priority of requirements..."

**RAL 的灵活扩展性**：团队可根据实际需要构建针对特定模型的工作流（如机器学习项目建模），核心框架不变。

---

#### 第 3 层：文档层 (DL) — 自动写需求规格文档

| 智能体 | 干什么 |
|--------|--------|
| 文档撰写智能体 | 把 RAL 产出的 UML 模型 + PL 的结构化需求 + 原始需求，合并撰写成 SRS 文档 |

**关键设计**：
- 文档模板可配置：小项目用轻量格式，大项目补齐 IEEE 830 所有章节
- 如果知识库缺某部分内容，该章节留空并标注——不乱编
- Prompt："You are a professional SRS generator. Generate the SRS using existing UML models, requirements decomposition tables, and original requirements..."

---

#### 第 4 层：验证生成层 (VGL) — 产出可确认的原型

| 智能体 | 干什么 |
|--------|--------|
| 原型生成智能体 | 根据需求规约生成原型（论文用视频原型），供用户确认"这是不是你要的东西" |

可插多种原型生成方式——写代码的智能体、模型转代码工具、或其他方案。

---

### 4.3 贯穿全流程：追溯矩阵

追溯矩阵不是智能体角色，而是**贯穿四层的基础设施**。每层处理完后自动存入关键信息：

```
用户原始需求 v1.0
    │
PL 处理  → 存入：结构化需求 ID + 需求类型 + 优先级 + 版本号
    │
RAL 处理 → 存入：UML 模型元素 → 需求条目的映射关系
    │
DL 处理  → 存入：文档版本号
    │
VGL 处理 → 存入：原型版本号
```

**需求变更时怎么工作**（以"用户登录"加"生物识别登录"为例）：
1. 新需求进来 → PL 分类为「功能需求，高优先级」→ 关联到原有「权限验证」子需求 → 版本号 v1.0 → v1.1
2. 追溯矩阵通知 RAL：Actor 扩展「生物传感器」、UseCase 新增「生物登录」继承「标准登录」
3. 追溯矩阵通知 DL 和 VGL：重新生成文档和原型
4. 全程记录变更影响面（本次影响 3 个元素）→ 准确率 85%，耗时 < 5 分钟，比手动维护降低约 80% 成本

---

### 4.4 共享基础设施：消息池 + 四个外部模块

所有智能体在一个共享消息池中交换信息。消息池包含五个组件：

| 组件 | 是啥 | 作用 |
|------|------|------|
| 背景信息 (Context) | 项目背景描述 | 帮智能体理解具体任务场景 |
| 系统提示 (SysPrompt) | 群组共享的目标描述 | 告诉所有智能体"咱们要干什么" |
| 消息 (Message) | 智能体之间交换的中间结果 | 协作通信的基础 |
| 工具 (Tools) | 外部可调用程序 | 排序算法、代码生成器等，智能体按需调用 |
| 知识 (Knowledge) | 特定领域知识 | 弥补通用 LLM 对特定领域的不了解 |

## 5. 数据与实验

**技术栈**：AgentScope 框架 + DeepSeek-R1 (1.5B/7B/14B) + Ollama 本地部署，温度设为 0.5

**数据集**：Wang 等人评估 ChatGPT UML 建模能力的公开案例，共 86 个建模元素。用例模型含 5 参与者 + 6 用例 + 12 关系；类图含 15 类 + 17 属性 + 14 方法；顺序图含 6 对象 + 21 条消息

**实验设计（4 组）**：

| 实验 | 对比什么 | 怎么做 |
|------|---------|--------|
| 实验 1 | 输入形式 | 结构化需求 vs 非结构化（原始文本）需求 → 看对建模准确率的影响 |
| 实验 2 | 模型规模 | 1.5B vs 7B vs 14B 的 DeepSeek-R1 → 看参数量对效果的影响 |
| 实验 3 | 温度参数 | 0.5 vs 1.0 vs 1.5 → 看温度对需求工程任务的影响 |
| 实验 4 | 需求变更 | 两组共 6 名硕士生，两轮迭代模拟，产生 15 条新需求 |

**评价指标**：
- 建模准确率：Actor / UseCase / Relationship / Class / Attribute / Function 各元素识别准确率
- 过度生成率
- Uni-REPM 需求过程评估指标（OS/PM/PM.CM/RA/QA 各维度，L1-L3 级）

---

## 6. 主要结论

### 正面结果
- 多智能体方法**可以**完成需求工程任务，在多数 Uni-REPM 指标中达到或超过 L2 标准
- 需求变更场景下，模型自动更新**有效**，准确率达 85%，自动化耗时 < 5 分钟，比手动维护降低约 80% 成本
- 14B 模型可以胜任 UML 建模，温度 0.5 效果最优（需求工程是严谨任务，不宜高温度）

### 关键负面发现（对 ChatREQ 非常重要）

| 发现 | 数据 | 原因 |
|------|------|------|
| 去掉原始文本后 Actor 识别崩溃 | 结构化需求 Actor 准确率 **0%**，非结构化 **66.7%** | 智能体缺乏原始语义上下文，推理时找不到"谁是执行者" |
| Relationship 识别始终失败 | 结构化和非结构化输入均为 **0%** | 知识库中关系样本过于稀缺 |
| 小模型完全不可用 | 1.5B 和 7B 模型无法完成 UML 建模 | 只有 14B 才显著可用 |

**核心论点**：智能体集群缺乏背景知识和上下文时，推理准确率大幅下降 → **知识增强不是锦上添花，是必需的**。

---

## 7. 局限

- 仅在单一数据集（86 个元素）上完成，学生模拟而非真实工业场景
- 未涉及金融合规、医疗监管等复杂领域
- Relationship 识别准确率为 0，是方法核心短板
- 只在 DeepSeek-R1 上测试，未验证其他 LLM
- 原型生成只到"视频原型"，未到可运行代码

## 8. 与 ChatREQ 对比分析

### 8.1 整体架构对比

| 维度 | ChatREQ | MA-ARP |
|------|---------|--------|
| 架构形式 | **串行流水线**：Interviewer → Knowledge → Analyst → Specification → Reviewer → Archivist | **分层并行**：PL → RAL → DL → VGL，层内多智能体同时干活 |
| 分工逻辑 | 按 **RE 流程阶段** 分（谁问 → 谁分析 → 谁写 → 谁检查） | 按 **建模子任务** 分（谁识别 Actor → 谁识别用例 → 谁识别关系） |
| 协作方式 | 上一个 Agent 输出 = 下一个 Agent 输入（串行传递） | 共享消息池，各智能体各自取需要的（并行+灵活） |
| 知识注入 | 独立 Knowledge Agent 集中检索注入 | 每层每个智能体各自调 Knowledge 模块 |
| 追踪机制 | Archivist Agent 手动记录 | 追溯矩阵基础设施自动记录 |
| 质量检查 | **有** — Reviewer Agent 5 维度检查 | **无** — 靠人工在结尾确认 |
| 原型验证 | **无** — Reviewer 通过就结束 | **有** — VGL 生成原型让用户确认 |

### 8.2 角色逐对对比

| ChatREQ 角色 | MA-ARP 对应 | 异同 |
|-------------|------------|------|
| **Interviewer** | 人工过程（需求工程师 + 用户沟通） | ChatREQ 把追问也交给 AI；MA-ARP 认为 AI 追问可能问不到点子上，保留人工 |
| **Knowledge** | Knowledge 外部模块 | ChatREQ 是独立 Agent 主动检索；MA-ARP 是被动模块，各 Agent 自己调 |
| **Analyst** | RAL 整层（10+ 个识别智能体） | ChatREQ 一个 Agent 全包 8 要素；MA-ARP 按模型元素拆成 N 个小 Agent 并行 |
| **Specification** | DL 层文档撰写智能体 | 功能相似。MA-ARP 多了可配置文档模板 |
| **Reviewer** | **无** | **ChatREQ 独有优势**。MA-ARP 缺少自动化质量门禁 |
| **Archivist** | 追溯矩阵 | ChatREQ 是一个 Agent 手动记录；MA-ARP 是基础设施自动记录，更可靠 |

### 8.3 关键差异总结

**ChatREQ 比 MA-ARP 好的地方**：
- Reviewer Agent 做独立质量检查，MA-ARP 完全没有
- 人机协同嵌入流程中间（Interviewer 追问），不只是头尾

**MA-ARP 比 ChatREQ 好的地方**：
- 追溯矩阵自动记录，比靠一个 Agent 手动记更可靠
- 层内并行协作，效率更高
- VGL 原型反馈闭环，ChatREQ 到 Reviewer 就停了
- 四类外部模块分类清晰（Prompt/Context/Tools/Knowledge）

## 9. ChatREQ 可以怎么改进

### 改进 1：Analyst 拆成并行子智能体

**现状问题**：ChatREQ 的 Analyst 一个角色做全部 8 个意图要素拆解（目标、角色、对象、动作、约束、场景、隐含假设、待澄清）。一个 Agent 同时处理 8 种要素容易漏——尤其隐含假设和待澄清问题最容易跳过。

**改进方案**：拆成 3 个子智能体并行：

```
原始需求
    │
    ├── 角色-目标识别 Agent   → 谁要干什么、为什么
    ├── 动作-场景识别 Agent   → 操作什么、什么上下文
    └── 假设-澄清识别 Agent  → 没说的事、该问什么
    │
    └── 汇总合并 → 完整 8 要素
```

**依据**：论文实验证明拆开比全包好——Actor 识别准确率 66.7%，Relationship 在缺乏分角色支持的情况下直接归零。

---

### 改进 2：Archivist 从 Agent 角色升级为自动基础设施

**现状问题**：Archivist 是一个 Agent，手动记录追踪链条。Agent 可能遗漏、记错，追踪链断了后续变更就没法定位影响面。

**改进方案**：每个 Agent 完成工作后自动写追踪记录：

```
Interviewer 输出    ──→ 自动存入：请求 ID + 原始表达 + 澄清问答
Knowledge 注入      ──→ 自动存入：知识条目 ID → 请求 ID
Analyst 拆解        ──→ 自动存入：要素 → 知识条目 → 请求 ID
Specification 生成   ──→ 自动存入：规约条目 → 要素
Reviewer 检查       ──→ 自动存入：检查结果 → 规约条目
```

需求变更时自动标出所有受影响条目。不再依赖 Archivist Agent 个人能力。

---

### 改进 3：引入共享消息池，防止信息衰减

**现状问题**：纯串行传递，链条越长信息衰减越严重。Specification 写规约时需要 Interviewer 的原始澄清对话，但中间过了 Knowledge 和 Analyst 两层，关键信息可能已经变形或丢失。

**数据支撑**：论文表 1——只用结构化需求（去掉原始文本），Actor 准确率从 **66.7% 直接崩到 0%**。

**改进方案**：每个 Agent 的输入 = 标准上游输出 + 可按需查询消息池中任意历史记录。比如 Specification 写规约时可以同时翻 Interviewer 的原始对话和 Knowledge 注入的知识条目。

---

### 改进 4：增加原型验证反馈环

**现状问题**：到 Reviewer 检查通过就结束。但规约质量通过了 ≠ 用户真满意，缺一个用户确认环节。

**改进方案**：在 Reviewer 之后加一步轻量文本原型（用户故事地图 / 验收场景列表），让用户确认。有反馈就回到 Interviewer 重新追问。论文证明这种闭环让迭代效率提升约 80%。

---

### 改进 5：规定人工介入边界

**现状问题**：Interviewer 自动追问没有轮数上限，可能越问越多、越问越偏。

**改进方案**：最多追问 2-3 轮，超过就标记"需人工介入"。MA-ARP 在头尾保留人工的做法证明这个边界有价值。

---

### 优先级总结

| 改什么 | 优先级 | 理由 |
|--------|--------|------|
| Analyst 拆子智能体 | **高** | 论文有数据支撑，直接提升分析质量 |
| Archivist → 自动基础设施 | **高** | 论文证明自动追踪优于角色追踪 |
| 共享消息池防衰减 | **中** | Actor 0% 是直接警示 |
| 原型验证反馈环 | **中** | M08 才有原型，前期不急 |
| 人工介入边界 | **低** | 锦上添花 |
| **Reviewer 保留不动** | **不该改** | 这是 ChatREQ 比 MA-ARP 强的核心差异点 |
