# 需求工程基础概念

## 1. 需求获取（Requirements Elicitation）

- **含义**：通过访谈、观察、文档分析等方式，从利益相关者那里收集需求信息。关键不是"用户说了什么"，而是"用户真正需要什么"。
- **与 ChatREQ 的关系**：Interviewer Agent 本质上就是在做需求获取——针对用户的自然语言表达，挖掘转换成真正的工程需求

## 2. 需求分析（Requirements Analysis）

- **含义**：对获取到的需求进行分类、拆解、冲突检测，识别出目标(项目最后的要达成的业务)、角色(系统的参与者)、对象(系统管控的实体）、、约束(落地的一些限制)和场景(用户使用场景)等要素。
- **与 ChatREQ 的关系**：Analyst Agent 负责需求要素分析，系统化的拆解

## 3. 需求规约（Requirements Specification）

- **含义**：将分析后的需求编写为结构化、清晰、可检查的文档。常见形式包括用户故事、用例、功能需求列表。
- **与 ChatREQ 的关系**：Specification Agent基于Analyst拆解的要素，负责生成结构化需求规约，是整个方法流程的核心输出物。

## 4. 需求验证（Requirements Validation）

- **含义**：检查需求是否正确反映了利益相关者的真实意图，是否完整、一致、明确、可验证。回答"我们写了正确的需求吗"。
- **与 ChatREQ 的关系**：Reviewer Agent 专门做需求验证，从多个维度检查需求质量(完整性，可行性，一致性，无歧义)

## 5. 需求管理（Requirements Management）

- **含义**：对需求进行版本控制、变更追踪、影响分析和状态管理，确保需求变更可追溯。
- **与 ChatREQ 的关系**：Archivist Agent 负责记录追踪关系——原始意图、知识条目、生成需求之间的链接，绑定三者。

## 6. 利益相关者（Stakeholder）

- **含义**：与系统有利益关系的任何人，包括用户、客户、开发人员、运维人员、监管机构等。不同利益相关者的需求角度不同，甚至互相冲突。
- **与 ChatREQ 的关系**：ChatREQ 的起点就是利益相关者的原始自然语言表达——这些表达往往不完整、模糊或有隐含假设，是我们利益需求的提炼的起点。

## 7. 功能性需求（Functional Requirement）

- **含义**：描述系统必须"做什么"的需求。例如"用户可以预约实验室""系统在冲突时给出提示"。
- **与 ChatREQ 的关系**：规约生成的核心内容。没有这个大模型就无从下手，做出的工能可能就会偏离你的核心需求

## 8. 非功能性需求（Non-Functional Requirement）

- **含义**：描述系统"做得怎么样"的需求，包括性能、安全性、可用性、可靠性、可维护性(做出来的一个东西效果怎么样)
- **与 ChatREQ 的关系**：通用大模型最常遗漏的就是非功能性需求。

## 9. 用户故事（User Story）

- **含义**：敏捷开发中最常用的需求形式，模板为"作为<角色>，我希望<功能>，以便<价值>"。强调用户视角和业务价值。
- **与 ChatREQ 的关系**：Specification Agent 可以输出用户故事格式。通过上游Analyst拆解出来的角色，目标，场景等，来嵌入这个角色，功能，价值

## 10. 用例（Use Case）

- **含义**：描述系统与参与者之间交互序列的建模方式，包含基本流程、备选流程和异常流程，比用户故事更详细(就是把用户故事拆解的更详细，更像是一个完整的说明书+应对各种意外情况的说明书)
- **与 ChatREQ 的关系**：Specification Agent先用 Analyst 给的素材生成简短用户故事，把用户故事展开细化，拆解出完整操作步骤、异常情况，生成完整用例文档，相当于先写一句摘要，再扩写成完整详细操作说明书。

## 11. 验收标准（Acceptance Criteria）

- **含义**：判断需求是否被满足的具体条件，必须可测试。例如"当用户输入错误密码 3 次后，账户锁定 15 分钟"。
- **与 ChatREQ 的关系**：Reviewer Agent来检查你这和需求，做完算不算合格，达不达得到标准

## 12. 需求优先级（Requirements Prioritization）

- **含义**：在有限资源下决定需求实现顺序的方法。常用 MoSCoW 法（Must/Should/Could/Won't）和 Kano 模型。
- **与 ChatREQ 的关系**：利息相关者这有利益冲突先做谁的

## 13. 需求冲突（Requirements Conflict）

- **含义**：两个或多个需求存在矛盾，无法同时满足。例如"响应时间 < 100ms"与"全数据加密"可能冲突。
- **与 ChatREQ 的关系**：大模型生成的代码可能会出现此类问题，Reviewer Agent 的一致性检查就是针对这个问题去解决。

## 14. 需求追踪矩阵（Requirements Traceability Matrix, RTM）

- **含义**：记录需求从来源到实现完整追踪关系的表格。正向追踪：需求→设计→代码→测试。反向追踪：代码→需求。关联所有需求工程阶段的追踪矩阵，支持动态更新和变更影响分析
- **与 ChatREQ 的关系**：Archivist Agent 记录的追踪链（原始表达→知识→澄清→需求→检查结果）就是一种简化版 RTM。

## 15. 需求变更管理（Requirements Change Management）

- **含义**：对需求变更的提出、评估、批准、实施和验证进行系统化管理。关键是在理解影响范围后再决策(防止系统失控)。
- **与 ChatREQ 的关系**：是先交给Analyst,重新拆解意图，看是否跟原先的意图违背，在按照原先的顺序一次往下工作

## 16. 业务规则（Business Rule）

- **含义**：定义或约束业务的具体规则，独立于软件实现。例如"每个学生最多预约 3 个实验室""超过 1000 元需审批"。
- **与 ChatREQ 的关系**：Knowledge需要去录入相关的专门的行业内容。

## 17. 领域模型（Domain Model）

- **含义**：领域模型 = 把现实业务提炼出：有哪些事物（实体）、事物有什么特征（属性）、事物之间是什么关系。
- **与 ChatREQ 的关系**：Knowledge Agent 搜集零散领域知识,汇总加工成领域模型；

## 18. 模糊性（Ambiguity）

- **含义**：需求存在多种合理解释。例如"系统应快速响应"——"快速"对用户是 100ms，对运维是 3 秒。是需求质量最常见的问题。
- **与 ChatREQ 的关系**：Reviewer Agent 要能发现模糊词，去追问用户问题，消除模糊的表述

## 19. 完整性（Completeness）

- **含义**：需求覆盖了所有必要方面——功能、角色、约束、场景、异常、接口。完整性是相对的，取决于系统边界和分析深度。
- **与 ChatREQ 的关系**：Reviewer Agent 评审完整性，检查是否漏功能 / 约束 / 异常。

## 20. 可验证性（Verifiability）

- **含义**：需求能否被测试、检查或确认。可验证的需求包含明确标准或可观察行为。反例："系统应易于使用"。
- **与 ChatREQ 的关系**：Specification Agent 编写规约时，把定性描述转化为可量化、可验证指标，Reviewer Agent筛查不可验证的需求描述，退回修订，模糊不可验证内容回溯 Interviewer，让用户补充量化验收标准。
