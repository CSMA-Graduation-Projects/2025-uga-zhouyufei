# 智能需求工程概念

## 1. 智能需求工程（Intelligent Requirements Engineering, IRE）

- **含义**：将人工智能技术（尤其是大语言模型）应用于需求工程各阶段——获取、分析、规约、验证、管理——以增强人工需求工程师的能力。不同于传统手工 RE，IRE 利用 LLM 的自然语言理解和生成能力来辅助或自动化 RE 子任务。
- **与 ChatREQ 的关系**：ChatREQ 本身就是一个 IRE 框架——用"大模型 + 需求工程知识 + 多智能体协作"把用户散乱的模糊自然语言需求转化为高质量的结构化规约。取代了原来客户说的自然语言需要用人力来记录，来理解的这么一个过程。


## 2. 意图要素分析（Intent Element Analysis）

- **含义**：将用户的原始自然语言表达拆解为结构化意图要素的过程。通常包含 8 个字段：目标（业务目的）、角色（系统参与者）、对象（系统管控实体）、功能动作、约束（落地限制）、场景（使用上下文）、隐含假设（用户没说但认为理所当然的）、待澄清问题。
- **与 ChatREQ 的关系**：Analyst Agent 核心能力就是意图要素拆解——不是简单分类，而是根据用户的需求，去拆解成这8个字段。


## 3. 意图偏移（Intent Drift）

- **含义**：大模型在生成需求或代码时，输出内容偏离用户原始意图的现象。具体表现：未经确认就扩展功能范围、遗漏隐含约束、替换用户明确指定的方案、添加用户未要求的"炫酷功能"。是 IRE 中最核心的质量问题之一。
- **与 ChatREQ 的关系**：Reviewer Agent就专门负责来管控这些，负责纠错ai在工作时擅自添加，修改的功能


## 4. 大模型需求失败模式（LLM RE Failure Patterns）

- **含义**：通用大模型在"裸用"（无知识注入、无角色分工、无检查规则）场景下执行需求工程任务时，系统性地出现的典型失败类型。常见模式包括：意图偏移、约束遗漏、角色混淆、场景缺失、未经确认扩展、不可验证表述、不可追踪、需求冲突等。
- **与 ChatREQ 的关系**：裸模型会出现大量繁琐问题，所以req一拆为六，拆分出不同的agent


## 5. 知识增强需求生成（Knowledge-Enhanced Requirements Generation, KERG）

- **含义**：在 LLM 生成需求时，将结构化的领域知识、需求工程知识或项目上下文知识注入 Prompt，使模型输出更完整、更一致、更符合领域规范。与通用 RAG（检索增强生成）的区别在于知识类型更聚焦——不是通用知识库检索，而是针对 RE 任务的特定知识。
- **与 ChatREQ 的关系**：Knowledge Agent 负责知识检索和注入，帮助ai写出符合行业规范的的需求


## 6. 多智能体需求协作（Multi-Agent RE Collaboration）

- **含义**：将需求工程流程拆分为多个专门的智能体角色，每个角色承担一个 RE 子任务，通过结构化的协作流程串接起来。典型角色分工：访谈（提问澄清）→ 知识注入 → 要素分析 → 规约生成 → 质量检查 → 追踪归档。
- **与 ChatREQ 的关系**：req设计至少 6 个智能体角色（Interviewer、Knowledge、Analyst、Specification、Reviewer、Archivist），定义各自的职责、输入输出和协作接口，相比于单模型"一步到位"容易出错，拆成多角色各自专注一个子任务效果更好。


## 7. 需求质量自动检查（Automated Requirements Quality Checking）

- **含义**：用 AI Agent（而非人工）对生成的需求规约进行质量评审，从 5 个维度逐条检查：完整性（是否遗漏角色/约束/异常）、一致性（需求之间是否冲突）、明确性（是否有模糊词）、可验证性（是否能设计测试）、可追踪性（是否能追溯到来源）。
- **与 ChatREQ 的关系**：Reviewer Agent 负责此任务，就是拿着这套 5 项检查标准，挨个审核 Specification Agent生成出来的需求文档。


## 8. Prompt 驱动的需求规约（Prompt-Driven Requirements Specification）

- **含义**：通过精心设计的 Prompt 模板来引导 LLM 执行需求工程任务，而非让用户随意对话。好的 RE Prompt 模板包含：角色设定（你是一个需求分析专家）、任务目标、知识注入区域、输出格式约束、质量自查要求。
- **与 ChatREQ 的关系**：贯穿 ChatREQ 全流程的基础设施。每个 Agent 背后都是一套可复用的 Prompt 模板，就是ai的一个岗位准则，靠着这个让ai不乱干活


## 9. 需求追踪自动化（Automated Requirements Traceability）

- **含义**：利用 AI 自动记录和维护需求从原始表达到最终规约的完整追踪链，包括中间所有的澄清问答、知识引用和质量检查结果。不同于传统 RTM（手工维护、易过时），自动追踪链随每次迭代自动更新。
- **与 ChatREQ 的关系**：Archivist Agent 负责追踪归档记录的链条是：原始表达 → 澄清问答 → 知识注入条目 → 意图要素分析 → 生成规约 → 质量检查结果。这一链条既支撑"这条需求从哪来"的可追溯性，也为后续变更影响分析提供依据。这是一条可以追溯的线路，以后改任何需求，方案，都依靠这一条线路作为依据


## 10. 人机协同需求澄清（Human-AI Collaborative Clarification）

- **含义**：AI Agent 不替代人做需求决策，而是通过主动发现模糊点和信息缺口，向人类利益相关者提出具体的、可回答的澄清问题（不能是"请详细说明你的需求"这样的笼统问题），由人类提供领域判断和最终确认。人在回路中（Human-in-the-loop），AI 做分析师，人做决策者。
- **与 ChatREQ 的关系**：Interviewer Agent 的核心交互模式。它从意图要素分析中发现缺失的角色、空白的约束、模糊的表述，然后生成针对性追问。把模糊的需求变成，去向人类提问，然后让人类来排版
