# 发明专利目标

本文件用于说明 ChatREQ 方向可沉淀的发明专利交底目标。本训练周期不表述为已经申请成功，而是逐步整理可用于专利交底的技术问题、技术方案、方法流程、系统组成和实施例。

## 1. 建议专利名称

- 一种基于知识增强与多智能体协作的智能需求生成与检查方法及系统
- 一种面向需求意图保持的大语言模型智能需求工程方法及系统

## 2. 技术领域

本方案属于软件工程、需求工程、自然语言处理和智能软件开发辅助技术领域，具体涉及一种基于需求工程知识、领域知识和多智能体协作的智能需求生成与检查方法及系统。

## 3. 背景问题

软件需求工程需要从利益相关者的自然语言表达中识别真实意图，并形成结构化、可验证、可追踪的软件需求。现有通用大语言模型能够生成较流畅的需求文本，但在实际需求工程任务中可能出现意图理解不足、约束遗漏、角色混淆、场景缺失、未经确认内容扩展、需求不可验证和不可追踪等问题。

## 4. 现有方法不足

现有方法可能存在以下不足：

- 仅将需求生成视为文本生成任务，缺少需求工程规范约束；
- 缺少对用户原始意图和隐含信息的系统分析；
- 缺少需求工程知识、领域知识和项目上下文的有效组织；
- 缺少多个需求工程子任务之间的协作机制；
- 缺少对生成需求质量的自动检查和人工复核流程；
- 缺少从原始表达到生成需求之间的追踪记录。

## 5. 要解决的技术问题

本方向拟解决的技术问题包括：

1. 如何从用户原始需求表达中识别目标、角色、对象、动作、约束、场景和隐含假设；
2. 如何组织需求工程知识、领域知识和项目上下文，并在需求生成过程中发挥作用；
3. 如何设计多智能体协作流程，使不同智能体分别完成需求访谈、知识注入、需求分析、规约生成、质量检查和归档；
4. 如何检查生成需求的完整性、一致性、明确性、可验证性和可追踪性；
5. 如何记录原始意图、知识条目、生成需求和质量检查结果之间的追踪关系。

## 6. 核心技术方案

核心技术方案包括：

- 构建需求意图要素分析模块，用于解析原始需求表达；
- 构建知识组织模块，用于管理需求工程知识、领域知识和项目上下文；
- 构建多智能体协作模块，用于协调访谈、知识注入、分析、规约、检查和归档；
- 构建需求规约生成模块，用于生成结构化需求条目；
- 构建需求质量检查模块，用于发现遗漏、冲突、模糊、不可验证和不可追踪问题；
- 构建追踪归档模块，用于记录意图、知识、需求和检查结果之间的关系。

## 7. 方法流程

1. 接收用户原始需求表达和项目上下文。
2. 识别需求意图要素，包括目标、角色、对象、动作、约束、场景和隐含假设。
3. 根据需求类型检索相关需求工程知识、领域知识和项目上下文知识。
4. 由 Interviewer Agent 生成澄清问题，并记录待确认信息。
5. 由 Analyst Agent 结合知识和澄清结果形成需求分析结构。
6. 由 Specification Agent 生成结构化需求规约。
7. 由 Reviewer Agent 按质量维度检查生成需求。
8. 根据检查结果修订需求规约。
9. 由 Archivist Agent 生成追踪记录和归档清单。

## 8. 系统组成

系统可包括：

- 原始需求输入模块；
- 需求意图分析模块；
- 知识库管理模块；
- 知识检索与注入模块；
- 多智能体协作调度模块；
- 需求规约生成模块；
- 需求质量检查模块；
- 追踪关系管理模块；
- 实验记录与结果导出模块。

## 9. 有益效果

与简单需求文本生成方式相比，本方案预期具有以下效果：

- 提高对用户原始意图的保持能力；
- 减少需求遗漏、约束丢失和未经确认扩展；
- 提升需求文本的完整性、一致性、明确性、可验证性和可追踪性；
- 使需求生成过程更加透明，便于复查和修改；
- 支持将需求工程知识和领域知识显式用于生成和检查；
- 便于形成论文实验材料和专利实施例。

## 10. 具体实施例

一个实施例可以围绕课程管理系统、在线问诊系统、校园预约系统或电商订单系统展开。以校园预约系统为例：

1. 用户输入“学生可以预约实验室，老师要能审核”。
2. 系统识别出角色包括学生、教师和管理员，动作包括预约、审核和管理。
3. 系统发现原始表达缺少预约时间、实验室容量、冲突处理、审核时限和取消规则。
4. Knowledge Agent 注入预约业务、权限控制和需求可验证性相关知识。
5. Interviewer Agent 提出澄清问题。
6. Specification Agent 生成结构化需求，包括功能需求、约束和验收条件。
7. Reviewer Agent 检查是否存在不可验证、角色不清或场景缺失问题。
8. Archivist Agent 记录每条生成需求对应的原始表达、知识条目和澄清问题。

## 11. 需要学生逐步整理的材料

- 原始需求表达案例；
- 需求意图要素分析表；
- 大语言模型失败案例；
- 需求工程知识条目；
- 知识与失败类型映射表；
- 多智能体角色与流程设计；
- 需求质量检查样例；
- 原型运行说明和示例；
- 论文预研报告；
- 发明专利交底草稿。

## 12. 专利目标说明

专利不是简单登记一个软件系统，而是要提炼“方法流程、知识组织方式、多智能体协作机制、需求质量检查机制”等技术方案。学生在整理材料时，应重点说明技术问题、技术手段和技术效果之间的对应关系。
