# ChatREQ解决方案：知识增强与多智能体协作的智能需求工程方法

本文件提出 ChatREQ 的解决方案框架。该框架用于指导本科科研训练中的案例分析、知识整理、实验设计、智能体流程设计、原型验证、论文预研和专利交底。

## 1. 总体方案

ChatREQ 方案的基本思路是：以用户原始需求表达为输入，先识别其中的目标、角色、对象、功能动作、约束、场景、隐含假设和待澄清问题；再引入需求工程知识、领域知识和项目上下文；通过多智能体协作完成需求追问、知识注入、需求分析、规约生成、质量检查和过程归档；最终输出结构化、可验证、可追踪的软件需求。

方案流程可以概括为：

1. 收集用户原始需求表达；
2. 分析需求意图要素和缺失信息；
3. 检索相关需求工程知识、领域知识和项目上下文；
4. 通过 Interviewer Agent 生成澄清问题；
5. 通过 Analyst Agent 完成需求要素分析；
6. 通过 Specification Agent 生成结构化需求规约；
7. 通过 Reviewer Agent 检查需求质量；
8. 通过 Archivist Agent 记录追踪关系；
9. 通过实验评价完整性、一致性、明确性、可验证性和可追踪性。

## 2. 需求工程知识库

需求工程知识库用于保存可被模型和智能体使用的规范性知识。知识条目不只是概念摘抄，而应能够服务具体任务。

知识库可包含：

- 需求获取知识：访谈问题、澄清策略、利益相关者识别方法；
- 需求分析知识：目标分析、角色分析、场景分析、约束分析、冲突识别；
- 需求规约知识：用户故事、用例、功能需求、非功能需求、验收标准；
- 质量检查知识：完整性、一致性、明确性、可验证性、可追踪性；
- 需求管理知识：版本管理、变更记录、追踪矩阵、影响分析。

每条知识应至少包含：知识内容、知识类型、知识来源、适用任务、可解决的失败类型和预期作用。

## 3. 领域知识与项目上下文

领域知识用于约束需求生成的业务合理性，项目上下文用于约束需求生成的系统边界。两类知识都可以减少模型无依据扩展和遗漏关键约束。

领域知识示例：

- 业务角色及权限；
- 常见业务流程；
- 数据对象与字段含义；
- 合规要求和隐私要求；
- 异常场景和边界条件。

项目上下文示例：

- 系统目标和范围；
- 已有功能和待开发功能；
- 用户群体；
- 技术约束；
- 历史需求和变更记录；
- 项目术语表。

## 4. 多智能体协作流程

多智能体角色应对应明确的需求工程子任务，不能只是简单命名。

| 智能体 | 需求工程子任务 | 输入 | 输出 | 主要作用 |
| --- | --- | --- | --- | --- |
| Interviewer Agent | 需求追问与意图澄清 | 用户原始表达、已知上下文 | 澄清问题、待确认事项 | 发现模糊、不完整和隐含信息 |
| Knowledge Agent | 知识检索与知识注入 | 需求表达、任务类型、知识库 | 相关知识条目、适用约束 | 为分析和生成提供依据 |
| Analyst Agent | 需求要素分析 | 原始表达、澄清结果、知识条目 | 目标、角色、对象、动作、约束、场景分析 | 将自然语言表达转化为分析结构 |
| Specification Agent | 结构化需求规约生成 | 需求要素、知识约束、上下文 | 功能需求、非功能需求、用户故事、验收条件 | 生成规范化需求文本 |
| Reviewer Agent | 需求质量检查 | 需求规约、原始表达、评价标准 | 质量问题、修改建议、风险标注 | 检查完整性、一致性、明确性、可验证性和可追踪性 |
| Archivist Agent | 追踪关系记录 | 原始表达、知识条目、生成需求、检查结果 | 追踪表、过程记录、归档清单 | 保留从意图到需求的证据链 |

一个基本协作流程为：

1. Interviewer Agent 读取原始需求并提出澄清问题；
2. Knowledge Agent 根据需求场景提供相关知识；
3. Analyst Agent 结合澄清结果和知识完成需求要素分析；
4. Specification Agent 生成结构化需求；
5. Reviewer Agent 根据质量维度检查需求；
6. Specification Agent 根据检查意见修订需求；
7. Archivist Agent 记录原始表达、知识、需求和修改意见之间的追踪关系。

## 5. 需求质量检查机制

需求质量检查机制用于判断生成需求是否达到需求工程基本质量。检查维度包括：

- 完整性：是否缺少角色、对象、动作、约束、场景或异常处理；
- 一致性：需求内部、需求之间、需求与上下文之间是否冲突；
- 明确性：是否存在模糊词、歧义表述或无法统一理解的描述；
- 可验证性：是否能形成测试条件、验收标准或检查规则；
- 可追踪性：是否能追溯到原始表达、知识条目和澄清记录；
- 意图保持：是否引入了未经确认的新目标或新功能。

质量检查结果应包含问题位置、问题类型、原因说明、修改建议和人工复核结论。

## 6. 实验验证机制

实验验证应采用小规模、结构清晰、可复查的方式进行。建议包含以下实验：

- 普通生成与知识增强生成对比；
- 单一模型角色与多智能体协作对比；
- 有质量检查与无质量检查对比；
- 不同知识类型对需求质量的影响分析；
- 生成需求与人工分析结果的一致性比较。

每组实验应记录输入、使用知识、协作设置、输出结果、评价指标、人工分析和失败原因。

## 7. 原型模块验证

原型模块应服务于研究验证，而不是单纯页面展示。最小可运行原型可以只实现一个核心流程，例如：

- 输入用户原始需求表达；
- 自动识别需求意图要素；
- 引入若干知识条目；
- 生成结构化需求规约；
- 输出需求质量检查结果；
- 保存追踪记录。

原型必须有运行说明、示例输入输出和截图，并说明它验证了哪个研究问题。

## 8. 论文与专利转化路径

论文预研转化重点：

- 需求工程中的意图偏移问题分析；
- 需求意图槽位或需求要素分析框架；
- 知识增强的需求生成机制；
- 多智能体协作的需求工程流程；
- 需求质量评价与案例实验。

专利交底转化重点：

- 方法流程；
- 知识组织方式；
- 多智能体协作机制；
- 需求质量检查机制；
- 需求意图、知识条目和需求规约之间的追踪机制。
