# tasks 任务包目录

`tasks/` 是本仓库最核心的任务包目录。学生不是泛泛“学习”或“参与项目”，而是按任务包推进 ChatREQ 科研训练。每个任务包都要有明确问题、明确输入、明确步骤、明确产物、明确存放路径和明确验收标准。

没有产物、不在仓库归档、不能说明对论文或专利支撑作用的任务，不视为有效科研训练。

## 1. 任务包列表

| 任务编号 | 任务名称 | 主要目标 |
| --- | --- | --- |
| M01 | 方向理解与基础概念 | 理解 ChatREQ 方向和智能需求工程基本概念 |
| M02 | 需求意图案例收集 | 收集并分析用户原始需求表达中的意图问题 |
| M03 | 大模型需求工程失败案例分析 | 分析通用大语言模型在需求工程任务中的失败现象 |
| M04 | 需求工程知识条目整理 | 整理可用于 ChatREQ 的知识条目和映射关系 |
| M05 | 知识增强需求生成实验 | 验证知识增强对需求生成质量的影响 |
| M06 | 多智能体角色与流程设计 | 设计多智能体角色和协作流程 |
| M07 | 需求质量检查实验 | 验证需求质量检查机制 |
| M08 | ChatREQ 原型模块验证 | 完成最小可运行原型或实验验证模块 |
| M09 | 论文预研材料整理 | 将前期材料整理为论文预研材料 |
| M10 | 发明专利交底材料整理 | 将方法机制整理为发明专利交底草稿 |

## 2. 推进方式

每个任务包都应按文件中的固定结构推进。建议学生在开始任务前先回答：

- 本任务解决哪个研究问题；
- 本任务需要哪些输入材料；
- 本任务完成后必须提交哪些文件；
- 这些文件应放在哪里；
- 如何判断达到完成标准；
- 本任务如何支撑论文和专利目标。

## 3. 产物要求

每个任务包的产物必须进入仓库对应目录。阶段报告放入 `reports/`，案例和数据放入 `datasets/`，知识条目放入 `knowledge/`，智能体设计放入 `agents/`，实验记录放入 `experiments/`，原型材料放入 `prototype/`，最终成果放入 `outcomes/`。

## 4. 检查要求

任务完成后，应至少检查：

- 文件是否存在；
- 内容是否完整；
- 数据、案例或实验记录是否可复查；
- 是否满足数量要求；
- 是否说明对论文目标的支撑；
- 是否说明对发明专利目标的支撑；
- 是否还有明显空泛表述。
